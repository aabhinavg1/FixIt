Every program needs to **make decisions**. At some point, the program must choose **which piece of code should run depending on a condition**.

:::caution For example, a program may need to decide:
:::

* whether debugging logs should be printed
* whether a feature should be enabled or disabled
* whether the code should run on Windows or Linux

These decisions are part of everyday software development.

In C and C++, there are **two different ways to implement such conditional behavior**:

1. **Preprocessor conditional compilation** using directives like `#if`, `#ifdef`, and `#ifndef`.
2. **Runtime conditional branching** using statements like `if`, `else`, and `switch`.

:::important
At first glance, both approaches appear similar because they both control **which code path is used**. However, they operate at **very different stages of a program's lifecycle**.
:::

To understand this better, let us consider a common real-world example.

Suppose we want to print **debug logs only in debug builds**, but completely remove them from release builds to keep the binary smaller and faster.

There are multiple ways to implement this. For this discussion, we will focus on **two approaches**:

1. Using a **runtime condition with `if`**
2. Using **compile-time conditional compilation with `#if` and macros**

Both approaches may look similar in the source code, but they behave very differently.



:::tip Implementation using `if` (Runtime Decision)
:::

```c
#include <stdio.h>

int main() {
    int debug = 1;

    if (debug) {
        printf("Debug log: initializing system\n");
    }

    printf("Program running\n");

    return 0;
}
```

In this version, the condition is evaluated **when the program runs**.

Both branches of the `if` statement are compiled into the program, and the CPU decides **at runtime** whether the debug message should be printed.


:::tip Implementation using Macro and `#if` (Compile-Time Decision)
:::

```c
#include <stdio.h>

#define DEBUG 1

int main() {

#if DEBUG
    printf("Debug log: initializing system\n");
#endif

    printf("Program running\n");

    return 0;
}
```

In this case, the decision is made **before compilation**, during the **preprocessing stage**.

The preprocessor evaluates the condition and **removes the unused code entirely**. If `DEBUG` is `0`, the debug log statement never reaches the compiler and does not exist in the final binary.



This example highlights the core difference:

* `#if` decides **whether certain code should exist in the compiled program**
* `if` decides **which code path should execute when the program runs**

Although both approaches solve similar problems, they operate at **different stages of the software build and execution process**.

:::tip Understanding this distinction is especially important for:

* systems programmers
* compiler engineers
* embedded developers
* GPU driver developers
* performance engineers
:::
because the choice between `#if` and `if` can affect several aspects of the final program, including:

* **binary size**
* **compiler optimizations**
* **runtime branching behavior**
* **instruction cache efficiency**

Even though `#if` and `if` may look similar in the source code, they serve **very different purposes in the lifecycle of a program**.

In the next sections, we will explore **how each mechanism works internally and when one should be preferred over the other**.

## Section 2: Where `#if` and `if` Fit in the Compilation Process

To fully understand the difference between `#if` and `if`, we first need to understand **how a C/C++ program is transformed into a running executable**.
I have already published article on it [how toolchain](https://www.compilersutra.com/docs/compilers/sourcecode_to_executable/) and [compiler](https://www.compilersutra.com/docs/compilers/IntroductionToCompilers/) works.

A typical C/C++ program passes through several stages before it becomes a runnable program.

***Classical Compilation Flow****

```python
Source Code > Preprocessor > Compiler > Assembler > Linker > Executable
```


Each stage performs a different task in converting source code into a working program.

* The **Preprocessor** processes directives like `#include`, `#define`, and `#if`.
* The **Compiler** analyzes the source code and generates intermediate or assembly code.
* The **Assembler** converts assembly instructions into machine code.
* The **Linker** combines multiple object files and libraries into a final executable.
* The **Loader** loads the executable into memory and starts the program.

In this article we mainly focus on the **first two stages**, because this is where the difference between **`#if` and `if` becomes important**.


**The Preprocessing Stage**

The **preprocessor** runs before the compiler begins compiling the program.

Its job is to process special directives that start with `#`.

Some common preprocessor directives include:

* `#include`
* `#define`
* `#if`
* `#ifdef`
* `#ifndef`

During this stage, the preprocessor performs several tasks:

* expanding macros
* including header files
* removing comments
* evaluating conditional compilation directives

Consider the following example:

```c
#define DEBUG 0

#if DEBUG
printf("Debug log\n");
#endif
```

If `DEBUG` is set to `0`, the preprocessor removes the `printf` line completely. The compiler **never sees this code**.

After preprocessing, the effective source code might look like this:

```c
int main() {
    return 0;
}
```

In other words, `#if` controls **what code is passed to the compiler**.


**The Compilation Stage**

Once preprocessing is finished, the compiler begins the **actual compilation phase**.

At this point:

* macros have already been expanded
* conditional compilation has already been resolved
* only the remaining source code is compiled

Now the compiler processes normal C/C++ statements such as:

```c
if (debug) {
    printf("Debug log\n");
}
```

Unlike `#if`, an `if` statement **cannot remove code during compilation**. Both branches remain part of the compiled program.

The compiler generates instructions that evaluate the condition **during program execution**.



**Key Difference**

The difference between these two mechanisms can be summarized clearly:

| Mechanism | Stage         | What it Controls                   |
| --------- | ------------- | ---------------------------------- |
| `#if`     | Preprocessing | Whether code exists in the program |
| `if`      | Runtime       | Which code path executes           |

Because of this behavior, `#if` is commonly used for:

* platform specific code
* enabling or disabling debug logs
* compile-time feature flags
* cross-platform builds

On the other hand, `if` is used for **normal program logic** that depends on runtime values.

## Section 3: What Happens When the Preprocessor Encounters `#if`


To understand the real impact of `#if`, it helps to see **what the preprocessor actually does before the compiler runs**.

The preprocessor reads the source file and processes all directives that start with `#`. One of its tasks is to **evaluate conditional compilation directives such as `#if`, `#ifdef`, and `#ifndef`**.

Based on the result of the condition, the preprocessor **either keeps or removes portions of the code** before the compiler ever sees it.

Consider the following example:

```c
#include <stdio.h>

#define DEBUG 0

int main() {

#if DEBUG
    printf("Debug log enabled\n");
#endif

    printf("Program running\n");

    return 0;
}
```

At first glance, this program appears to contain two `printf` statements. However, the preprocessor evaluates the `#if` condition before compilation.

Since `DEBUG` is defined as `0`, the condition becomes false, and the debug statement is removed.

The code that actually reaches the compiler becomes:

```c
//expansion of stdio.h

int main() {

    printf("Program running\n");

    return 0;
}
```

Notice that the debug statement has completely disappeared.

This demonstrates an important property of `#if`:

`#if` **does not create a runtime branch**. Instead, it **removes or keeps code before compilation**.


:::caution Viewing Preprocessor Output
:::

Most compilers allow you to see the output produced by the preprocessor.

For example, using Clang:

```
clang -E example.c
``` 
>> it will dump everything on stdout terminal 

```python
clang -E example.c &> example.i
vim example.i  // you can use editor of your choice
```

The `-E` flag tells the compiler to **run only the preprocessing stage** and print the resulting source code.

This is extremely useful for understanding:

* how macros expand
* how header files are included
* how conditional compilation modifies the source code

When you run this command on the previous example, you will see that the debug statement **no longer exists in the preprocessed output**.



:::tip Why This Matters
:::
Because the removed code never reaches the compiler, it also means:

* the code **does not increase the binary size**
* the compiler **does not analyze or optimize it**
* the CPU **never executes it**

This is why `#if` is widely used in large codebases such as:

* operating systems
* embedded firmware
* compiler implementations
* GPU drivers
* cross-platform libraries

It allows developers to **compile different versions of a program from the same source code** depending on build configuration or target platform.


In the next section, we will explore real-world use cases of #if, and see how macro values are defined and controlled in large systems such as the Linux kernel and cross-platform libraries.

## Section 4: Real-World Use Cases of `#if` and Macro Configuration

In real software systems, conditional compilation is often used to **enable or disable parts of the program during the build process**.

This allows the same codebase to support multiple configurations such as:

* debug vs release builds
* platform-specific implementations
* architecture-specific optimizations
* feature toggles during development

Let us start with a simple program and then see how `#if` can control what code gets compiled.


**Base Program**

Consider the following simple program:

```c
#include <stdio.h>

int main() {

    printf("System starting...\n");

    printf("Program running\n");

    return 0;
}
```

Compile the program using **Clang**:

```bash
clang program.c
./a.out
```

Output:

```
System starting...
Program running
```

Now suppose we want to add **debug logging**, but we only want those logs in **debug builds**, not in release builds.


***Adding Debug Logging with `#if`***

We can control this behavior using a macro and conditional compilation.

```c
#include <stdio.h>

#define DEBUG 1

int main() {

    printf("System starting...\n");

#if DEBUG
    printf("Debug: initializing subsystem\n");
#endif

    printf("Program running\n");

    return 0;
}
```

Compile using **Clang**:

```bash
clang program.c
./a.out
```

Output:

```
System starting...
Debug: initializing subsystem
Program running
```

The directive `#if DEBUG` is evaluated by the **Clang preprocessor before compilation begins**.


**Disabling Debug Code**

Now suppose we change the macro value:

```c
#define DEBUG 0
```

The program becomes:

```c
#include <stdio.h>

#define DEBUG 0

int main() {

    printf("System starting...\n");

#if DEBUG
    printf("Debug: initializing subsystem\n");
#endif

    printf("Program running\n");

    return 0;
}
```

Compile with Clang:

```bash
clang program.c
./a.out
```

Output:

```
System starting...
Program running
```

Because the condition evaluates to false, the **preprocessor removes the debug line completely before the compiler sees the code**.

The compiler effectively receives:

```c
#include <stdio.h>

int main() {

    printf("System starting...\n");

    printf("Program running\n");

    return 0;
}
```

The debug statement **never reaches the compiler or the generated binary**.


:::important Changing Macro Values from the Build System
:::

In large systems such as the **Linux kernel**, macros are rarely hardcoded inside the source file. Instead, they are defined by the **build system**.

Using **Clang**, macros can be defined from the command line using the `-D` option.

Example:

```bash
clang -DDEBUG=1 program.c
```

The `-D` flag tells the **Clang preprocessor** to define a macro before processing the source file.

This is equivalent to inserting this line at the top of the program:

```c
#define DEBUG 1
```

This allows developers to **build different versions of the same program without modifying the source code**.

Example builds:

Debug build:

```bash
clang -DDEBUG=1 program.c
./a.out
```

Release build:

```bash
clang -DDEBUG=0 program.c
./a.out
```


:::caution Macro Redefinition Warning in Clang
:::

If the macro is defined **both in the source file and on the command line**, Clang will generate a warning.

Example program:

```c
#include <stdio.h>

#define DEBUG 0

int main() {

    printf("System starting...\n");

#if DEBUG
    printf("Debug: initializing subsystem\n");
#endif

    printf("Program running\n");

    return 0;
}
```

Compile using:

```bash
clang -DDEBUG=1 program.c
```

Clang warning:

```python
program.c:3:9: warning: 'DEBUG' macro redefined [-Wmacro-redefined]
#define DEBUG 0
        ^
<command line>:1:9: note: previous definition is here
#define DEBUG 1
```

This happens because the preprocessor sees two definitions:

```python
#define DEBUG 1   (from command line)
#define DEBUG 0   (from source file)
```

The later definition overrides the earlier one.



:::tip Using `#ifndef` to Avoid Redefinition
:::
To avoid this issue, most production systems use the following pattern:

```c
#ifndef DEBUG
#define DEBUG 0
#endif
```

Full example:

```c
#include <stdio.h>

#ifndef DEBUG
#define DEBUG 0
#endif

int main() {

    printf("System starting...\n");

#if DEBUG
    printf("Debug: initializing subsystem\n");
#endif

    printf("Program running\n");

    return 0;
}
```

Now Clang behaves correctly:

Compile without debug:

```bash
clang program.c
```

Compile with debug enabled:

```bash
clang -DDEBUG=1 program.c
```

The build system can now safely control the configuration without generating warnings.

This technique is widely used in:

* operating systems
* embedded firmware
* compilers
* cross-platform libraries

because it allows one codebase to support **multiple configurations and environments** using the build system.

## Section 5:  When to Use `if` Instead of `#if`

While `#if` is powerful for controlling what code gets compiled, it is **not always the right choice**. In many cases, a normal `if` statement is more appropriate.

The key rule is simple:

* Use **`#if` when the decision must be made during compilation**
* Use **`if` when the decision depends on values known only at runtime**

Let us look at some common situations.


:::tip 1. When the Condition Depends on Runtime Data
:::

If the decision depends on **user input, system state, or program variables**, you must use a normal `if` statement.

For example:

```c 
#include <stdio.h>

int main() {

    int age;

    printf("Enter age: ");
    scanf("%d", &age);

    if (age >= 18) {
        printf("Person is an adult\n");
    } else {
        printf("Person is not an adult\n");
    }

    return 0;
}
```

In this case, the value of `age` is **not known during compilation**.
It is provided by the user while the program is running.

Because of this, the decision must be made **at runtime**, which means `if` is required.

Using `#if` here would not work.


:::tip 2. When Code Should Always Exist but Behave Differently
:::

Sometimes both code paths should exist in the compiled program, but the program chooses one depending on conditions during execution.

Example:

```c
#include <stdio.h>

int main() {

    int debug;

    printf("Enter debug mode (1 = enable, 0 = disable): ");
    scanf("%d", &debug);

    if (debug) {
        printf("Debug mode enabled\n");
    }

    printf("Program running\n");

    return 0;
}
```

Here the debug behavior can be turned on or off **while the program runs**, without recompiling the program.

This flexibility is something `#if` cannot provide.


:::tip 3. When Code Readability Matters
:::

Overusing `#if` can make code **harder to read and maintain**, especially when many nested conditions appear.

Example of confusing code:

```c 
#if PLATFORM_LINUX
    #if DEBUG
        printf("Linux debug\n");
    #endif
#endif
```

In large projects, too many conditional compilation blocks can make the code difficult to understand.

Because of this, developers often prefer **runtime conditions** unless compile-time control is truly required.


### Summary

The choice between `#if` and `if` depends on **when the decision needs to be made**.

| Situation                         | Preferred Approach |
| --------------------------------- | ------------------ |
| Compile-time configuration        | `#if`              |
| Platform-specific builds          | `#if`              |
| Feature flags during build        | `#if`              |
| User input or runtime data        | `if`               |
| Program behavior during execution | `if`               |

Understanding this difference helps developers choose the **right mechanism for the right situation**.

## Section 6: Performance Considerations: #if vs if

To understand the performance difference between compile-time and runtime decisions, we created a **simple benchmark program**.

The goal of this testcase is to compare three approaches that perform the **same logical operation**:

1. **Macro with `#if` (compile-time decision)**
2. **Runtime conditional using `if`**
3. **Function-based implementation**

All three implementations run the same loop with **100 million iterations** and perform a simple computation.


**Test Program**

The following program implements the three approaches:

```c
#include <stdio.h>
#include <time.h>

#define DEBUG_MACRO 0
#define ITERATIONS 100000000

volatile long sum = 0;

/* Macro version */
void macro_version() {

    for (long i = 0; i < ITERATIONS; i++) {

#if DEBUG_MACRO
        sum += i;
#endif

    }
}

/* Runtime if version */
void runtime_version(int debug) {

    for (long i = 0; i < ITERATIONS; i++) {

        if (debug) {
            sum += i;
        }

    }
}

/* Function version */
void debug_function(long i) {
    sum += i;
}

void function_version(int debug) {

    for (long i = 0; i < ITERATIONS; i++) {

        if (debug) {
            debug_function(i);
        }

    }
}

int main() {

    int debug;
    clock_t start, end;

    printf("Enter debug value (0 or 1): ");
    scanf("%d", &debug);

    printf("\nRunning macro version...\n");

    start = clock();
    macro_version();
    end = clock();

    printf("Macro version time: %f seconds\n",
           (double)(end - start) / CLOCKS_PER_SEC);

    printf("\nRunning runtime if version...\n");

    start = clock();
    runtime_version(debug);
    end = clock();

    printf("Runtime if version time: %f seconds\n",
           (double)(end - start) / CLOCKS_PER_SEC);

    printf("\nRunning function version...\n");

    start = clock();
    function_version(debug);
    end = clock();

    printf("Function version time: %f seconds\n",
           (double)(end - start) / CLOCKS_PER_SEC);

    return 0;
}
```

**What This Testcase Does**

The program runs **three different implementations** of the same operation.

1. Macro Version

```
#if DEBUG_MACRO
    sum += i;
#endif
```

This code is processed by the **preprocessor**.

Since `DEBUG_MACRO` is set to `0`, the preprocessor removes the code completely **before compilation**.

The compiler effectively sees:

```
for (...) {
}
```

This means:

* no runtime condition
* no branch instruction
* almost no work inside the loop


2. Runtime `if` Version

```
if (debug) {
    sum += i;
}
```

In this version, the condition must be evaluated **during program execution**.

The CPU performs the following steps for every iteration:

1. load `debug`
2. compare value
3. perform branch decision
4. execute or skip code

Since the loop runs **100 million times**, the CPU performs **100 million condition checks**.


3. Function Version

```
if (debug) {
    debug_function(i);
}
```

Here the logic is placed inside a function.

Modern compilers may **inline small functions**, meaning the function call may disappear and the code may be optimized directly into the loop.

Because of this, the function version can sometimes perform **better than a simple runtime branch**.


**Running the Benchmark**

Compile the program with optimization:

```
gcc -O2 performance_test.c -o performance_test
```

Run it:

```
./performance_test
```

Example outputs:

:::tip Case 1: debug = 1
:::

```
Running macro version...
Macro version time: 0.000010 seconds

Running runtime if version...
Runtime if version time: 0.047409 seconds

Running function version...
Function version time: 0.033322 seconds
```

:::tip Case 2: debug = 0
:::
```
Running macro version...
Macro version time: 0.000006 seconds

Running runtime if version...
Runtime if version time: 0.046981 seconds

Running function version...
Function version time: 0.033319 seconds
```


**Observations**

The results highlight several important behaviors.

**Macro (`#if`)**

* condition evaluated **before compilation**
* code removed entirely
* fastest execution

**Runtime `if`**

* condition evaluated **during execution**
* branch instruction executed each iteration
* additional runtime overhead

:::tip Function Approach
:::

* may be **inlined by compiler**
* often optimized better than raw branch logic

:::tip Key Takeaway
:::
This testcase demonstrates the core difference between compile-time and runtime conditions:

| Approach | When Evaluated | Result                            |
| -------- | -------------- | --------------------------------- |
| `#if`    | Preprocessing  | Code removed before compilation   |
| `if`     | Runtime        | Condition checked every execution |
| Function | Runtime        | May be optimized or inlined       |

In performance-critical systems such as **operating systems, compilers, and embedded firmware**, removing unnecessary runtime branches using `#if` can improve efficiency.

However, excessive use of macros can also make code harder to maintain, so developers must choose the right tool for the right situation.


| Aspect               | `#if`                  | `if`              |
| -------------------- | ---------------------- | ----------------- |
| Binary Size          | Smaller                | Larger            |
| Compilation          | Removed before compile | Compiled normally |
| Runtime Cost         | None                   | Branch cost       |
| Optimizer Visibility | Not visible            | Fully visible     |

## section 7: Best Practices: When to Use `#if` vs `if`

Both `#if` and `if` are important tools in C and C++ programming, but they solve **different types of problems**. Choosing the right one depends on **when the decision needs to be made** — during compilation or during program execution.

Understanding this distinction helps developers write code that is **efficient, maintainable, and easier to debug**.


:::tip Use `#if` for Compile-Time Decisions
:::
`#if` should be used when a decision must be made **before the program is compiled**.

In these cases, the preprocessor determines whether certain code should exist in the final program.

Typical scenarios include:

* **Platform-specific code**

```c
#ifdef _WIN32
    printf("Running on Windows\n");
#else
    printf("Running on Linux\n");
#endif
```

* **Debug or release builds**

```c
#define DEBUG 1

#if DEBUG
printf("Debug logging enabled\n");
#endif
```

* **Compile-time feature flags**

```c
#if ENABLE_EXPERIMENTAL_FEATURE
run_new_algorithm();
#endif
```

Using `#if` in these situations allows the compiler to **remove unused code completely**, which can reduce binary size and improve performance.



:::tip Use `if` for Runtime Decisions
:::

The `if` statement should be used when the condition depends on **values that are only known while the program is running**.

Examples include:

* user input
* configuration values
* program state
* runtime system conditions

Example:

```c
int debug;

scanf("%d", &debug);

if (debug) {
    printf("Debug mode enabled\n");
}
```

Here the program cannot decide the behavior during compilation because the value is only known at runtime.


:::note Avoid Overusing Macros
:::

Although macros are powerful, excessive use of conditional compilation can make code harder to maintain.

For example:

```c
#if FEATURE_A
    process_feature_a();
#elif FEATURE_B
    process_feature_b();
#elif FEATURE_C
    process_feature_c();
#endif
```

When many macros are combined, the code becomes difficult to read, test, and debug.

Because of this, many large projects try to **limit the use of conditional compilation** to configuration boundaries such as:

* platform support
* build configurations
* optional modules



 **General Guideline**

A simple rule that many experienced developers follow is:

| Situation                            | Preferred Approach |
| ------------------------------------ | ------------------ |
| Platform or architecture differences | `#if`              |
| Build-time feature configuration     | `#if`              |
| Debug or release builds              | `#if`              |
| User input or runtime values         | `if`               |
| Program logic                        | `if`               |



:::tip Final Thoughts
:::

Compile-time and runtime conditions serve **different roles in software development**.

* `#if` helps control **how the program is built**
* `if` controls **how the program behaves while running**

Using them correctly allows developers to build software that is **efficient, portable, and easier to maintain**, especially in large system-level projects such as operating systems, compilers, and embedded firmware.

## Section 8: When to Use Neither `#if` Nor `if`

While `#if` and `if` are powerful tools for conditional logic, there are situations where **neither approach is the best choice**. In many cases, better design techniques such as **configuration files, polymorphism, function pointers, or compile-time templates** can lead to cleaner and more maintainable code.

Overusing conditional logic—especially nested `#if` blocks—can make programs difficult to understand and maintain.

Let us look at a few scenarios where avoiding both `#if` and `if` may be beneficial.



:::tip 1. When Configuration Should Be External
:::
Sometimes program behavior should be controlled **without recompiling the program**.

Instead of using compile-time macros or runtime `if` statements scattered throughout the code, configuration can be placed in a **configuration file or environment variable**.

Example:

```c
#include <stdio.h>
#include <stdlib.h>

int main() {

    char *debug = getenv("DEBUG");

    if (debug && atoi(debug) == 1) {
        printf("Debug mode enabled\n");
    }

    printf("Program running\n");

    return 0;
}
```

Here the program behavior can be controlled **without modifying or recompiling the source code**.

This approach is commonly used in:

* production systems
* cloud services
* distributed applications



:::tip 2. When Design Patterns Are More Appropriate
:::
Large systems often avoid excessive conditional logic by using **design patterns**.

Instead of writing code like this:

```c
if (type == 1) {
    process_type_a();
} else if (type == 2) {
    process_type_b();
}
```

A cleaner approach may be to use **function pointers or polymorphism**.

Example using function pointers:

```c
void process_type_a() { printf("Type A\n"); }
void process_type_b() { printf("Type B\n"); }

void (*handler)(void);

int main() {

    int type = 1;

    if (type == 1)
        handler = process_type_a;
    else
        handler = process_type_b;

    handler();

    return 0;
}
```

This technique is widely used in:

* operating systems
* driver frameworks
* plugin architectures



:::tip  3. When Compile-Time Logic Should Use Modern C++ Features
:::
In modern C++, compile-time decisions are often implemented using **`constexpr if`** or **template specialization** instead of preprocessor macros.

Example using `constexpr if`:

```cpp
template<typename T>
void process(T value) {

    if constexpr (sizeof(T) == 4) {
        printf("32-bit type\n");
    } else {
        printf("Other type\n");
    }

}
```

Unlike macros, this approach:

* respects type safety
* integrates with the compiler's type system
* improves readability



 ###### Summary

Sometimes the best solution is **not to use conditional compilation or runtime branching at all**.

Alternative approaches may include:

| Scenario                | Better Alternative                           |
| ----------------------- | -------------------------------------------- |
| Runtime configuration   | Configuration files or environment variables |
| Multiple behavior types | Function pointers or polymorphism            |
| Compile-time type logic | `constexpr if` or templates                  |

Choosing the right design approach helps keep code **cleaner, more maintainable, and easier to extend**.

## Section 9: Conclusion

The difference between `#if` and `if` can be summarized succinctly:

> **`#if` decides whether code exists. `if` decides which code runs.**

Understanding this distinction is essential for:

* performance engineering
* embedded systems
* compiler design
* GPU drivers
* operating systems development

Mastering these concepts allows developers to write **predictable, efficient, and maintainable low-level code**.
